# **v1.0 ルーレットアニメーションシステム仕様書**

## **1\. 概要**

本ドキュメントは、Raspberry Pi Zero 2 WH（以下、RPi Zero）上で動作するDiscordボット用のルーレットアニメーション生成システムの仕様を定義する。

リソース制約の厳しい環境下で、ユーザー体験（UX）を損なわず、かつサーバーへの負荷を最小限に抑えた「時間分散レンダリング」方式を採用する。

## **2\. ハードウェア制約**

* **CPU:** Broadcom BCM2710A1 (クアッドコア, 1GHz)  
* **RAM:** 512MB LPDDR2 (極めてタイト)  
* **ストレージ:** SDカード (I/O Waitおよび摩耗を考慮)  
* **冷却:** ヒートシンクケース装着済み（サーマルスロットリング耐性あり）

## **3\. アニメーション・シーケンス設計**

全体の演出を4つのフェーズに分け、既製品（静的ファイル）と動的生成を組み合わせることで、生成待ち時間を「演出時間」の中に隠蔽する。

| フェーズ | アニメーション内容 | 形式 | 生成タイミング | 役割 |
| :---- | :---- | :---- | :---- | :---- |
| **Phase 1** | 超高速回転 | GIF (固定) | 事前 | 即時フィードバック。期待感の醸成。 |
| **Phase 2** | 若干減速 | GIF (固定) | 事前 | 停止への予兆。裏で停止GIFの生成時間を稼ぐ。 |
| **Phase 3** | 減速 | GIF (固定) | 事前 | 停止直前の溜め。 |
| **Phase 4** | **停止シーケンス** | **GIF (動的)** | **実行時** | **確定した出目（角度）に滑らかに停止する。** |

### **3.1 タイムライン詳細**

1. /roulette 受付完了後、ボットは即座に Phase 1 のGIFを送信。  
2. ボットのバックグラウンドプロセスが Phase 4 (停止) のレンダリングを開始。  
3. asyncio.sleep 等で同期を取りながら、メッセージを Phase 2 \-\> Phase 3 へ編集（Edit）。  
4. 最終的に生成が完了した Phase 4 に差し替え、ルーレットが止まる。

## **4\. 技術実装（RouletteCanvas）**

v1.1\_rl\_canvas\_anim.py に基づく実装仕様。

### **4.1 レンダリング方式**

* **アセット合成方式:** 背景、ホイール、玉を別々のレイヤーとしてメモリに保持。  
* **回転処理:** Pillow.Image.rotate (BICUBICリサンプリング) を使用。  
* **停止ロジック:** 三次関数イージング（Ease-out Cubic）を用いて、等比数列的に duration を増加させることで物理的な摩擦をシミュレートする。

### **4.2 最適化手法**

* **メモリ管理:** 各フレーム生成後に buf への書き出しを行い、不要なオブジェクトを参照解除する。  
* **I/O削減:** 全ての計算と合成はメモリ（または /dev/shm）上で行い、SDカードへの書き込みを避ける。

## **5\. セキュリティと信頼性**

* **リソース枯渇対策 (DoS防止):** \- 同時レンダリング数を制限（Semaphore等）。  
  * 連続リクエストに対するレートリミットの実装。  
* **可用性:** \- psutil によるメモリ監視を行い、閾値を超えた場合は静止画のみのフォールバックモードへ切り替える。  
* **データ整合性:** \- 乱数生成には secrets モジュールを使用し、暗号学的に安全な出目決定を行う。
