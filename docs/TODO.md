## 1. 非同期処理におけるデータ競合 (Race Condition)

### 現状の課題
`WalletManager` および `BetService` の経済処理において、以下の Read-Modify-Write パターンが使用されています。
1. `SQLiteStorage` から現在の残高をロード
2. メモリ上で計算（加算・減算）
3. `SQLiteStorage` に新しい残高を保存

Discord ボットは非同期（`asyncio`）で動作するため、複数のコマンドが同一ユーザーに対して同時に実行された場合（例：対人戦の終了とデイリーボーナスの取得が重なるなど）、ステップ1と3の間に別の処理が入り込み、データが上書きされるリスクがあります。

### 将来の改善策
- **SQLレベルでのアトミック更新**: `UPDATE users SET balance = balance + ? WHERE user_id = ?` のような、DB側で完結するアトミックな SQL クエリに移行する。
- **ロック機構の導入**: ユーザーIDごとの `asyncio.Lock` を導入し、同一ユーザーへの経済操作が直列化されるようにする。

## 2. デプロイ時の安全性向上

### 現状の課題
`deploy.py` は、リモートからの DB 同期が失敗した場合でも、ローカルにある古い（あるいは空の）DB をそのまま ZIP してアップロードする可能性があります。これにより、本番データが損失するリスクがあります。

### 将来の改善策
- **強制中断ロジック**: 同期失敗時にデプロイを ABORT する（実装済み）。
- [x] ブラックジャックのスプリット機能
  - [x] データ構造の刷新（マルチハンド対応）
  - [x] スプリットアクションの実装
  - [x] ホスピタリティ適用範囲の精緻化（手札単位）
  - [x] `BlackjackView` の複数手札表示対応
- [x] テストの完全な隔離
  - [x] `core/storage.py` でパスをハードコードせず、環境変数などで切り替えられるようにする。
  - [x] テスト実行時は常にテスト用一時DBを使用するように `pytest` の fixture (conftest.py) を整備する。

### 現状の課題
`SQLiteStorage` などがデフォルトで `data/discord_eco_sys.db` を参照しているため、テストで Mock や `tmp_path` を使い忘れると本番データに影響を与える可能性があります。

### 将来の改善策
- **環境変数の利用**: `APP_ENV=test` のときは自動的にメモリ内 DB または一時ディレクトリを使用するように強制する。

## 5. ドブモン (Dobumon) の血統・遺伝バランス

### 現状の課題と検討事項
- **寿命計算の見直し**: 禁忌血統（近親交配など）による短命化のペナルティを、減算ではなく「乗算（割合での短命化）」に切り替えることを検討する。
- **インフレ（Power Creep）への対応**: 遺伝シミュレーションの結果、10世代程度の交配でHP等のステータスが数万〜数百万規模へと指数関数的にインフレすることが確認されている。倍率ベースで寿命を短命化させれば、こうした巨大な育成値の個体であっても既存の計算式を大きく破綻させることなくバランス調整ができる可能性がある。
